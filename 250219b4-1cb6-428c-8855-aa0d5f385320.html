开始用笔记 蹄蹄子问答录

蹄蹄子要换用 Obsidian 了，然后化身问题宝宝。老鼠解答问题的同时也被卷了一下，目前打算好好整理一遍笔记了。挺好的，相互促进进步~

把蹄蹄子提过的问题（部分）记录一下，答案可能不是当时的答案，毕竟即问即答对于反应慢的老鼠来说难免疏漏，所以这里写下的是整理之后可能不那么丢脸的答案。

为什么用 Obsidian
  • 为了我的数据是我的。
  • 云端笔记要顾虑：
    • 网络问题
    • 收费政策变化
  • 隐性问题：
    • 数据是否可导出
    • 是否可以无损导出通用格式
    • 是否可以不受限制免费导出
  • 所以选择基于 Markdown 文件的 Obsidian
    • 格式通用
    • 无需导出
    • 各种 Markdown 编辑器都可以直接打开，甚至没有编辑器人脑渲染压力也不大。（但是要注意不能乱用私有语法）。
  • 我并不依赖 Obsidian，它对于我只是一款笔记管理工具，长文编辑我会打开 Typora，复杂编辑我会打开 VS Code，对外发布目前我用自写的网站生成器。
他们的笔记真好看，我可以拥有么？

Obsidian 的美化是非常有限且无限的。

  • 有限：基于 Markdown，而 Markdown 是非常简陋的。这注定了它适合输出，不适合观赏。
  • 无限：可以插入 HTML，可以修改 CSS，可以运行 JS，网页能做什么它能做什么，网页能怎么美化它能怎么美化。只要你会写，愿意去写。
那我复制过来不就行了

复制过来的代码当然是可以用的。甚至有人贴心的写成了插件、Dataview 的视图，Templater 的模板等。

但是，里面的内容总要换成自己的，替换内容是否简便，这不太确定，取决于这些作者是否足够贴心。而且你希望的结结果和他给出的总会有所不同（确实看图的时候我们会想要个一模一样的，但是用起来总会产生自己的想法），而你做出的微小变动会不会导致整个模板出问题呢？（这是很正常的现象）

举个例子，我们日常工作中下载的 PS 素材模板或者 Word 模板，或者 PPT 模板……有没有遇到过那种看起来很漂亮，但是到手里很难修改，一动全都乱掉的情况呢？这种问题在各种模板中都普遍存在，Obsidian 中也不能例外。

所以，如果你完全不懂，就只能做完形填空，丧失了很多灵活性。

可以先有一个漂亮的起始页么？

不可以，因为起始页是依托笔记而存在的。和笔记完全无关的……那不是起始页，而是功能页。一个合理的起始页应该是联通了你大部分（常用）笔记的入口，汇总了你希望快速获取的笔记相关的信息。

起始页可以先有一个空架子，然后随着笔记增加再往里填么？

笔记是在成长和变化的。起始页也会随着这种成长而变化。而一些查询要根据具体情况书写（笔记的格式），在还没有笔记的时候先制定出所有笔记的书写规范，听起来挺科学的，但实际操作起来可能是直接劝退的，所以我前期指定的都是大方向，基本原则，而具体细节都是在使用中不断完善的，这也是一种笔记成长的表现。所以，在还没有笔记的时候这些查询是没法写的。于是空架子就只剩下了一个样子，和直接插入一张图片的效果类似。

而且，既然笔记是成长的，起始页也是成长的，现在的空架子还没等填空可能就不适合你了，让起始页随着自己的需求而变化就好了。

何况，谁说只能有一个起始页，完全有多个满足不同需求的起始页。当然起始页也不是必须的，当你觉得需要一个汇总、中控的时候它才需要出现。

美化的悖论（老鼠补充）

不是反对美化，也不是说美化和实用必须完全对立，只是美化和使用之间常常存在着一些取舍的关联，理解这些关系，可以更好的决定每一个元素的去留，并且给自己充足的理由。

很多美化是用性能消耗换来的，如果你的设备足够好，确实可以忽略多数的问题，但是当在手机上打开的时候呢？毕竟，笔记先是实用的，所以总要考虑一些极限情况。

简单举例，我的起始页有一张随机头图，我喜欢，可以给我不同的心情。但是从网络中加载、渲染一张图片必然是比只渲染本地文字需要的时间和性能更高的。这很不起眼，但很多这种细节堆积下来就会明显拖慢软件了。电脑上还可以打开笔记之后就一直不关，避免冷启动。但手机上就很难做到这一点。

在其中寻找一个适合自己的平衡点，这是很必要的。

类似的问题不只是限于美化，还有功能，每一个功能的增加也是要靠性能换的，所以插件装不装，装哪个，也值得斟酌。确实，做个笔记还要考虑这么多很烦人啊！但渐渐发现很多事情都存在着类似的矛盾，都需要为它们寻找平衡点。除非，完全不美化，不增加功能……但是当你选择 Obsidian 而不是系统记事本的时候，不也是在这些问题上寻找的平衡么……

手机版编辑时底下有快速格式的工具条，电脑版没有……

电脑版可以安装一个叫做 cMenu 的第三方插件来实现。搜索时会看到一个叫做 Editing Toolbar 的插件，比较新，也比较强，但加入了一些非原生 markdown 的东西，这方面我比较排斥（因为我很注重语法的通用性，不能离开一些插件就不能用了），不过因为加入的肯定是 HTML 和 CSS，所以只是 markdown 不纯粹，倒也不会过度影响迁移。

我自己实际安装确认了一下，cMenu 日常就够用的，而且很简洁。

段落怎么首行缩进？

这就是一个 CSS 样式。在 Obsidian 中可以通过添加 CSS 片段（CSS Snippets）来实现。方法如下：

  • 打开自己笔记仓库所在文件夹（可以在左侧目录中找一个文件夹右键——在系统资源管理器中显示），找到一个叫做 .obsidian 的文件夹，进去
  • 如果没找到这个文件夹，记得打开显示隐藏文件再看看
  • 然后在其中的 snippets 文件夹（没有则自己新建）中新建一个 css 文件，比如叫做 custom.css，那这个文件的路径是这样的：.obsidian/snippets/custom.css
  • 在这个 css 文件中（没有编辑器就用记事本打开），粘贴下方代码，p 代表的是段落元素，text-indent 是说首行缩进距离，2em 大概就是两倍字符宽度
  • 保存好之后并没有任何效果，这是正常的。还要到 Obsidian 的设置——外观（拉到最底下）——CSS 代码片段中开启你刚才定义的样式。
p {
   text-indent: 2em;
}

第一次操作确实有点小麻烦，但以后对于样式的修改大概就修改这个文件就可以了。一些细节上的美化都可以通过这个方式进行。

怎么让代码五颜六色的

就这样指明代码块内的语言是 css（具体哪种语言）

```css
p {
    text-indent: 2em;
}
```

私有语法（老鼠补充）

一些插件会引入私有语法（格式），毕竟纯粹的 markdown 格式功能非常弱。这是合理的，但得适度。

如果是短期使用的数据，比如说最近的待办事项，或者能够确定一俩年后，自己肯定不再关心这些数据。那用怎样的私有语法都是没所谓的。因为这在这个插件的生命周期之内（如果今天开始这个插件就不再更新它大概也能够维持一两年的正常使用）

但如果这些内容，你希望长期有效，甚至等自己老了坐在摇椅上，说不定还要回顾一下。那就不能乱引入私有格式了，除非这种格式直接在文本状态下阅读，也具有很好的可读性。

放在一个稍长一些的时间尺度下，别说这些可能只是某个开发者心血来潮写的插件，即便 Obsidian 也很难说可以使用几年。毕竟我们也都换了几个笔记软件了。

所以从做笔记的最初就开始注意保持自己笔记语法的纯洁性（容易被其他同类软件完美兼容），能够给以后的自己少留一点麻烦。

不过这个细节可以按笔记区分，某篇笔记要纯洁，要保持兼容性；某篇笔记就很激进的享受各种插件的便利；某篇笔记只有一点点可读性很好的私有语法……但得自己了解并掌控这些。

⭐绝密（觉得老鼠絮叨了才可以看）

这只老鼠絮絮叨叨的，反反复复的，婆婆妈妈的讲的全是车轱辘话嘛，感觉他总是绕来绕去就绕回去了，全是一个味道，初时新奇，后来就没味道了。

因为我的讲解必然围绕着我对笔记的理解展开，我理解到什么程度，就讲到什么程度，不可能超脱出去的。我对笔记在意的点：

  • 笔记得是自己的，始终处于自己的掌控之中；
  • 数据得是自己的，不能过度依赖特定软件和插件；
  • 方法是自由的，目的是解决问题，所以笔记法不可能脱离开要解决的问题；
  • 笔记是成长的，留给它成长的空间，见证这种成长；
  • 让工具回归工具的本质，不神化，不贪多；
  • 编程（代码）是用来创造工具的工具，不要刻意把它孤立出来，就是个瑞士军刀，随身携带，会带来便利。
