0014 分享前的说明

  • 《遇见 Obsidian》系列笔记汇总目录
  • 如果觉得有用就支持一下作者：老鼠爱发电

前面大概把自己的思路、原则、用法说了一下。大家应该也对一些插件的工作方式有了了解。后面我主要就是分享我自己写的一些功能代码了。这里说明一些基础设定：

关于目录

  • Scripts 是脚本目录，js 文件放这里，Templater 设置中的脚本文件夹指向这里
  • Templater 是 Templater 的模板目录，Templater 设置中的模板文件夹指向这里
  • CSS 指的是“设置——外观——CSS 代码片段”中所指的路径 .obsidian/snippets

关于代码块

主要区分在语言标记：

  • 原生查询：query
  • Dataview：dataview
  • Dataview 的 JS 查询： dataviewjs
  • Templater 的模板代码直接写，不要放在代码块里

关于注释

正常使用是不需要注释的，而且不加注释性能还好一点，不过为了大家可以根据需求适当修改，我会尽量多写点。你可以删除注释，甚至压缩代码。

额外插件

还是回归到原则，关注笔记本身，不依赖平台和插件。所以上面这些主要都是查询和流程自动化。软件外观啥的我并不太在乎，但如果可以生成好看的符合 Markdown 语法的笔记，我绝对不介意。

还有一个值得讲的插件是 Tasks，也有自己的查询语法，针对任务查询，挺强的，语法也比较简单。不过和 Dataview 太多重叠了。我现在用它只是自动标记任务完成日期（大材小用了）

0013 真的快起来

  • 《遇见 Obsidian》系列笔记汇总目录
  • 如果觉得有用就支持一下作者：老鼠爱发电

Obsidian 的界面真的谈不上好，好多按钮，好多菜单，有时候想做什么还得先思考一下在哪里找。

好在可以用命令，搜一下。但某些很流程化的事情还是会让人烦躁，这时候就应该 QuickAdd 出场了。

它能做的事情很多，不过一个重点场景就是——添加笔记。

闪念

或者叫碎碎念，2333。

反正忽然想做个什么记录，也不是新建笔记，就便签，随手记。或者想到要做什么事情，加一条待办。

我只是想快速记录，记录上就好，我不想找到文件（或者新建，取名字），写入内容，再关闭掉。

QuickAdd 的 Capture 就是做这个事情的。可以调出一个输入框，输入内容回车，就保存到对应文件的对应位置（是的，可以设定具体位置），并且从始至终都没打开那个文件（也可以设定为添加后打开）。

怎么设置……懒得写了，你自己去查一下吧，其实自己看设置也挺容易探索的。

这里说个小技巧：最开始我也觉得 QuickAdd 的设定页面有点迷，但是在手机上用的时候就完全没有这种感觉，好像特别顺理成章，水到渠成。就是布局问题，所以在电脑上，把 Obsidian 界面拉窄一些再去设置 QuickAdd 就不那么迷了。

然后，QuickAdd 用的模板中变量得用 QuickAdd 方式书写，具体查看一下文档。

套用模板

比如我书写这个系列的内容时，文件都要放在同一个文件夹下，笔记元数据也差不多。我就用 QuickAdd 的模板功能，它能自己套模板新建文件，我就负责书写就好。一键开启新篇章。

这个功能用 Templater 也可以实现，不过还得自己写脚本，懒捏！

快捷键

电脑上设定快捷键调用操作，很方便，我懒得记太多，Ctrl+Q 唤起 QuickAdd，然后选择操作。

手机上面

手机上可以设置一个快捷动作，就是界面下拉，我设置为了 QuickAdd，虽然也可以设置为 QuickAdd 中的具体动作，但是我需求比较多，所以还是每次选一下吧。

脚本

可以实现多动作连续执行，可以在打开 Obsidiann 的时候执行，可以自己写脚本……

挺强的，但我用不了那么多，所以对于这个插件主要是上面的两种用途。

当然用 <%+ %> 或者 <%* %> 都是可以的，也可以和其他代码放在一起共用，总之，和日常使用 JS 区别不大。

有人说，看出来了啊，上一篇的代码就是和 JS 混合书写的，那现在是不是把上一篇的代码复制过来做个完形填空就好呢？

基本上是的，不过在用户脚本里没有 tp 这个对象，tp.date.now() 也就无从说起。怎么办呢？把 tp 当参数传回来就行了。

一般如果需要输出，函数负责返回内容，在模板中再根据具体调用方法进行输出。

有朋友说：你看这个大老鼠，又要夹带私货，试图教会我们 JS 了！

并不是啦，只是这种自由度的工具还是多少了解一点它的基本运行思路会用的比较顺畅。否则有一天你遇到了问题，然后自己去看看代码，发现 <%、<%+、<%* 三种情况，然后强迫症发作给统一了一下，然后看到 一个 xxx.js 的文件从来没用到过，就给删了……问题就越来越多了。

你说：我不管，别人写好我就用，出了问题不乱动！

我觉得像动态执行这种代码很少有人给直接写好吧，简单情况下只是一个 + 的区别，自己动手一下很合理。以及虽然模板可以写出交互式设定，但是参数太多的时候也不可能全都问一下的，所以自己稍微改改代码挺合理的。

比如：一个模板的输出文字有点大，你看到代码里有：font-size: 18px，很显然可以修改数字看看嘛，这不需要什么编程啊，懂代码啊。

我后面的内容也都是尽可能浅显的解说一下插件的基本运作思路，然后代码尽量多加注释，让大家有一些自行修改的余地。
